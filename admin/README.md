# Admin CMS

Decap CMS instance for managing site content. Built as a separate Vite bundle that outputs to `public/admin/`, served alongside the main SolidStart app.

## Dev Workflow

Two Vite servers run in parallel during development:

| Server     | Config                 | Port | Purpose            |
| ---------- | ---------------------- | ---- | ------------------ |
| SolidStart | `vite.config.ts`       | 5173 | Main app + proxy   |
| Admin      | `vite.config.admin.ts` | 5174 | Decap CMS with HMR |

`npm run dev` starts all four processes via `concurrently`:

```
vite dev                                             # SolidStart on :5173
vite dev --config vite.config.admin.ts               # Admin on :5174
decap-server                                         # Local Git backend, third party tool; output ignored
tsx scripts/generate-cms-config.ts --dev --watch     # Generates config.yml + watches content for .md changes
```

The SolidStart server proxies all `/admin` requests to port 5174 (HTTP + WebSocket). HMR is configured with `clientPort: 5173` so the browser opens its WebSocket back through the proxy rather than directly to 5174. A `301` redirect in `vite.config.ts` handles bare `/admin` → `/admin/` to avoid Vite's "public base URL" message.

## Build

Two independent Vite configs exist side by side:

| Config                 | Entry              | Output          |
| ---------------------- | ------------------ | --------------- |
| `vite.config.ts`       | SolidStart app     | `.output/`      |
| `vite.config.admin.ts` | `admin/index.html` | `public/admin/` |

The admin build resolves the `~` alias to `src/` so it can import shared model definitions directly from `src/models/`.

`npm run build:admin` builds the admin bundle. `prebuild` runs it automatically (after `generate:cms` and before `validate:content`).

## CMS Config Generation

`public/admin/config.yml` is **gitignored** and generated at startup from the collection registry. It is never hand-edited.

| Script             | Backend                                          | When                           |
| ------------------ | ------------------------------------------------ | ------------------------------ |
| `generate:cms:dev` | `local_backend: true` + `test-repo`              | `predev`, watched by `dev:cms` |
| `generate:cms`     | GitHub (`CMS_GITHUB_REPO` / `CMS_GITHUB_BRANCH`) | `prebuild`                     |

Required env vars for production builds (set in `.env`):

```
CMS_GITHUB_REPO=owner/repo
CMS_GITHUB_BRANCH=main
```

`dev:cms` generates `config.yml` at startup, then watches content folders for `.md` file changes (via Node's `fs.watch`) to regenerate the per-entry JSON files and `cms-manifest.json`. TypeScript model changes require restarting the process. The admin Vite server detects the `config.yml` write (via the `reloadOnConfigYml` plugin) and sends a full-reload. Model files are excluded from admin Vite's own watcher to prevent a race condition where the reload fires before the new `config.yml` is written.

## Collection Models as Single Source of Truth

Collections are defined once in `src/models/` using `defineCollection` and `fields` builders from `src/lib/cms/define`. The same definitions drive three things:

1. **`public/admin/config.yml`** — generated by `generate:cms` / `generate:cms:dev`. The `collections` section is built entirely from `collectionRegistry`.
2. **Content validation** — `npm run validate:content` (runs at prebuild) parses each `.md` file through the Zod schema and exits non-zero on any mismatch.
3. **Typed content fetcher** — `getCollection` / `getCollectionItem` in `src/lib/content.ts` parse content through the same schema and return typed results.

All collection fields are wrapped in `.optional()` at the `defineCollection` level. This means partially-written entries and CMS preview data with missing fields will not blow up the client — callers are expected to handle `undefined` values.

## Preview Templates

All collections get an iframe-based preview template registered automatically:

```ts
collectionRegistry.forEach((collection) => {
  CMS.registerPreviewTemplate(
    collection.name,
    buildIframePreviewComponent({ previewPath: collection.previewPath })
  );
});
```

`buildIframePreviewComponent` is a factory rather than a single component because Decap CMS preview templates receive only `entry` and CMS-provided props — there is no way to pass custom props at registration time.

### `previewPath`

Each collection definition carries a `previewPath: (slug: string) => string` function that maps a slug to the URL the iframe should load. It defaults to `(slug) => \`/${slug}\`` in `defineCollection`. Override it for collections whose entries live at a different path:

```ts
export const blogCollection = defineCollection({
  name: 'blog',
  // ...
  previewPath: (slug) => `/blog/${slug}`,
  fields: { ... },
});
```

### Iframe sizing

The preview component uses a `ResizeObserver` on the Decap-provided parent container to mirror its exact `clientHeight` onto the iframe, avoiding both the 150 px browser default and the slight overflow that `100vh` produces inside the CMS chrome.

## Live Preview

When an editor changes a field, Decap re-renders the preview component with a new `entry` prop. The component responds via a `useEffect` that:

1. Calls `entry.get("data").toJS()` to extract the raw field map from Decap's Immutable.js record.
2. Posts a `CmsPreviewMessage` (`{ type, slug, data }`) to the iframe's `contentWindow`.
3. Re-registers a `load` listener each effect run so newly navigated iframes also receive the initial data.

Inside the SolidStart iframe, `createCmsLiveContent` (`src/primitives/createCmsLiveContent.ts`) listens for these messages, validates `data` through the collection's Zod schema, and returns a reactive signal. Routes compose it with `??` over the async-fetched content:

```ts
const liveContent = createCmsLiveContent(pagesCollection, () => params.slug);
const content = () => liveContent() ?? page();
```

Live content takes priority over the fetched manifest so unsaved edits are visible immediately without a file write.

## Click-to-Edit Field Focus

When the user clicks a `[data-cms-field]` element in the SolidStart preview iframe, a `cms-field-focus` message travels up to the admin window, which then scrolls to and focuses the matching field in the Decap editor.

### Iframe nesting

Decap renders preview templates inside a `react-frame-component` `<Frame>`, creating three levels:

```
admin window  (admin.tsx listener, Decap editor DOM)
  └── Decap preview iframe  (IframePreview React component)
        └── SolidStart iframe  (our app, data-cms-field elements)
```

`postMessage` from the SolidStart iframe must target `window.top` to skip the middle frame and reach the admin window directly. `window.parent` only reaches the Decap preview iframe.

### Message flow

1. User clicks `[data-cms-field]` in the SolidStart iframe.
2. `setupCmsPreview` (`src/lib/utils/setupCmsPreview.ts`) calls `window.top.postMessage({ type: "cms-field-focus", fieldPath }, "*")`. `fieldPath` is a dot-separated path built from `CmsPathContext` + the clicked element's property name (e.g., `modules.0.content`).
3. The `message` listener in `admin.tsx` receives it and calls `navigateToField(fieldPath)`.

### `navigateToField`

Decap CMS has no public API for programmatic field focus. `navigateToField` walks the dot-separated `fieldPath` segment by segment against the admin document DOM:

```
keys = fieldPath.split(".")
searchSpace = document.documentElement

for each key:
  if numeric → find the nth list item container inside searchSpace, set searchSpace to it
  if string  → find label whose textContent includes "[key]" inside searchSpace
    if last segment → scrollIntoView + focus (4-step focusable search)
    if not last     → set searchSpace to the field's container element and continue
```

Labels are generated with a `[key]` suffix (e.g., `"Content [content]"`) by `defineCollection` and `fields.object`/`fields.list`, making them unambiguous when nested. Numeric segments handle list items by finding the nth child of the list container.

The 4-step focusable search at the leaf: `label[for] → getElementById` → `label.querySelector("input, textarea, [contenteditable]")` → `label.nextElementSibling.querySelector(...)` → `label.parentElement.nextElementSibling.querySelector(...)`. Covers Decap's string/text widgets (input is sibling of label) and markdown widgets (contenteditable lives in a separate container). Focus is delayed 500 ms to let `scrollIntoView` settle.

The listener lives in `admin.tsx` (not `IframePreview.tsx`) because `document.querySelectorAll` must run against the admin document — the Decap editor fields only exist there, not inside the preview iframes.

## Browser Dialog Patches

Two `window` methods are monkey-patched in `admin/browser-patches.ts`, imported before `CMS.init()`:

**`window.confirm`** — auto-accepts (returns `true`) the local backup recovery dialog ("A local backup was recovered for this entry, would you like to use it?"). This silences the prompt on every page load without discarding the backup.

**`window.addEventListener`** (dev only, guarded by `import.meta.env.DEV`) — swallows any `beforeunload` registration, preventing Chrome's "Changes you made may not be saved." navigation dialog during development.

## Environment Variables

`VITE_SERVER_URL` — prepended to all iframe `src` URLs and manifest fetch URLs. Defaults to `""` (same-origin). Set to `http://localhost:5173` in `.env` for local development so the admin (served on port 5174) can reach the SolidStart dev server.

## React and TypeScript

`admin/tsconfig.json` configures `jsx: "react-jsx"` with `jsxImportSource: "react"`, so `.tsx` files in the admin directory use the React JSX automatic transform. React itself is not an explicit dependency — it is available as a transitive dep of `decap-cms-app` and imported directly from `react`.
